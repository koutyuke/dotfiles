# 評価軸ガイド

依頼ごとに、結論を分ける軸だけを 3〜6 個選ぶ。ここにある軸を網羅的なチェックリストとして使わない。

## 軸の作り方

各軸を次の形にする。

| 項目 | 内容 |
| --- | --- |
| 名前 | 案の特徴ではなく、達成したい成果 |
| 重要度 | `高 / 中 / 低` または意思決定者が合意した重み |
| 方向 | 大きいほど良い、小さいほど良い、閾値を満たせばよい |
| 観測方法 | 数値、検証手順、具体的な状態 |
| 根拠の確度 | `高 / 中 / 低`。評価軸の点数とは分ける |

良い軸は、案が違えば評価が変わり、評価が変われば選択に影響する。差が出ない軸や選択に影響しない軸は外す。

## 代表的な軸

### 成果・有効性

- 見るもの: 要件達成率、利用者価値、業務成果、品質目標
- 具体化例: `主要ユースケースの成功率 99.9% 以上`

### 導入速度

- 見るもの: 最初の価値までの時間、完了までの期間、学習期間
- 具体化例: `2 週間以内に本番の 10% へ導入できる`

### 実装複雑性

- 見るもの: 変更範囲、依存関係、認知負荷、テスト難度
- 具体化例: `変更対象が 2 service 内に収まり、既存 API を壊さない`

### 保守性

- 見るもの: 変更容易性、責務分離、テスト容易性、属人化
- 具体化例: `通常の仕様変更を 1 人日以内で安全に反映できる`

### 信頼性・運用負荷

- 見るもの: 可用性、障害頻度、復旧時間、監視、手作業、on-call 負荷
- 具体化例: `月間 SLO 99.9%、復旧 30 分以内、定常手作業なし`

### セキュリティ・コンプライアンス

- 見るもの: 権限境界、秘密情報、監査可能性、規制、supply-chain risk
- 必須条件にする例: `PII を対象地域外へ保存しない`

### Total Cost of Ownership

- 見るもの: 初期費用、継続費用、人件費、移行・撤退費用
- 具体化例: `3 年間の費用と必要 FTE を同じ前提で比較する`

### 可逆性・ロックイン

- 見るもの: rollback、段階導入、データ移行、契約、固有 API 依存
- 具体化例: `1 営業日以内に旧方式へ戻せる`

### 既存資産・組織適合

- 見るもの: architecture、toolchain、team skill、support model、移行摩擦
- 具体化例: `既存の監視・CI・認証基盤をそのまま使える`

### 拡張性・将来余地

- 見るもの: 想定負荷、要件追加、組織拡大、標準への追従
- 具体化例: `利用量 10 倍でも再設計せず水平拡張できる`

## 技術設計でのシナリオ化

`性能`, `可用性`, `変更容易性` のような品質属性は、名前だけでは採点しない。可能なら次を定義する。

- 刺激: 何が起きるか
- 状況: どの環境・負荷で起きるか
- 応答: system がどう振る舞うべきか
- 測定値: latency、error rate、変更時間など

例: `通常負荷の 5 倍の burst 時も、read API の p95 latency を 300 ms 未満に保つ`。

## よくある重複

- `実装しやすさ` と `開発難度`
- `保守性` と `変更容易性`
- `導入速度` と、同じ工数を再度評価する `初期コスト`
- `拡張性` と、同じ将来要件を再度評価する `将来性`
- `運用負荷` と、運用人件費を含む `TCO`

重複を残す場合は、何を別々に測るのかを明示する。

## 使い分け例

- 緊急修正: `成果`, `導入速度`, `可逆性`, `障害リスク`
- 長期基盤: `成果`, `保守性`, `信頼性`, `TCO`, `拡張性`
- SaaS 選定: `要件適合`, `security`, `TCO`, `integration`, `lock-in`
- 移行方針: `移行リスク`, `downtime`, `可逆性`, `期間`, `TCO`
- architecture: 抽象的な軸名ではなく、上記の品質シナリオを優先する

## 軸にしないもの

- 根拠の確度: 案の価値ではなく評価の metadata として扱う。
- 必須条件: 加重平均へ入れず pass / fail で先に判定する。
- すべての案で同じ値: 比較に寄与しないため省く。
- 言い換えただけの総合評価: 個別軸と二重計上になるため省く。
